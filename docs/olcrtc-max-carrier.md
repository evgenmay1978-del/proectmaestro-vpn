# olcRTC carrier «MAX» — feasibility-исследование (ВЕРДИКТ: не осуществимо сейчас)

**Итог: MAX нельзя сделать carrier-ом olcRTC с доступными сегодня данными.** Это не вопрос
трудозатрат — упор в фундаментальные несовместимости модели и в неотреверсенный протокол.
Провайдер `max` жёстко заблокирован в коде (`olcMaxCarrierReady=false`, `MAX_READY=0`), пункт
в панели убран. Ниже — что выяснено и почему.

## Что такое MAX и его звонки

MAX (`max.ru`, Android-пакет `ru.oneme.app`) — мессенджер VK. Звонки на generic WebRTC
(`libjingle_peerconnection_so.so` в APK). Реальная архитектура звонков (по публичному реверсу
[`icyfalc0n/maxcalls`](https://github.com/icyfalc0n/maxcalls)) — трёхслойная:

1. **OneMe API** — `wss://ws-api.oneme.ru/websocket` (Origin `https://web.max.ru`). Требует
   **телефонной аутентификации** (opcode 17 VerificationRequest → SMS → 18 → ChatSync 19),
   затем `CallTokenRequest` (opcode 158) отдаёт call-token.
2. **Calls API** — `https://calls.okcdn.ru/fb.do` (это инфраструктура звонков **OK.ru/VK**!):
   `auth.anonymLogin` (по call-token из OneMe) → сессия; `vchat.startConversation` → отдаёт
   `turn_server`, `stun_server` и `endpoint` (WebSocket сигналинга).
3. **Signaling API** — кастомный WebSocket (Origin `web.max.ru`): `ServerHello`, `accept-call`,
   `transmit-data` (несёт полезную нагрузку сигналинга как непрозрачный `data: any`), `ping/pong`.

## Как устроен carrier в olcRTC (и почему MAX туда не ложится)

Бинарник `libolcrtc.so` собирается из внешнего `openlibrecommunity/olcrtc` (пин `OLCRTC_REF`).
Carrier = **provider** (auth-флоу, `internal/auth/<name>`) + **engine** (SFU-сигналинг на
**pion/webrtc**, `internal/engine/<name>`). Модель работы: exit-srv **присоединяется к комнате
по ссылке и держит её открытой**, клиент присоединяется в ту же комнату → туннель едет поверх.
Так работают все три существующих carrier-а:

| provider | engine | join-by-link | гостевой вход | заметка |
|----------|--------|:---:|:---:|---------|
| `telemost` | `goolom` (pion) | да | **аноним** | ссылка на комнату Яндекс.Телемоста |
| `wbstream` | `livekit` (pion) | да | гость (srv как аккаунт держит комнату) | bare room id |
| `jitsi` | `jitsi` (pion) | да | аноним | MUC/Jingle |

## Блокеры MAX (каждый — сам по себе стоп)

1. **Нет комнаты по ссылке / нет персистентного rendezvous.** Реверснутый API MAX умеет
   *начать* звонок авторизованным аккаунтом (`vchat.startConversation`) и *ответить* на входящий —
   но **не** «присоединиться к общей ссылке-комнате». Вся модель olcRTC (srv сидит в комнате,
   клиент заходит туда же) на MAX не воспроизводится: делить нечего.
2. **Только телефонная аутентификация, нет анонимного/гостевого входа.** Каждый участник (и
   exit-srv, и клиентское приложение на устройстве пользователя) должен быть залогинен в MAX по
   номеру телефона + SMS. Клиент — это приложение пользователя; нельзя предполагать, что там есть
   аккаунт MAX. У telemost/jitsi вход анонимный, у wbstream srv заходит как аккаунт, а клиент —
   гость в LiveKit-комнате; у MAX гостя нет вообще.
3. **In-call SDP/ICE обмен не отреверсен.** В `transmit-data` полезная нагрузка — `data: any`;
   формат оффера/ансвера и обмена ICE **не документирован** ни в maxcalls, ни где-либо ещё.
   Сам maxcalls реализует только auth + конверт сигналинга и использует **собственный** ICE-стек
   (`internal/ice`, не pion), причём лишь для режимов outgoing/incoming — рабочего join/media там
   нет. Чтобы получить формат, нужен живой звонок с реальным аккаунтом + перехват WebSocket-кадров
   сигналинга и разбор конверта — отдельный крупный RE-проект.
4. **Импеданс со стеком olcrtc.** Engine-ы olcrtc — pion/webrtc (`*webrtc.DataChannel`,
   `webrtc.TrackLocal`). У MAX/maxcalls — свой ICE/сигналинг-стек и SFU OK.ru vchat (okcdn.ru).
   Переходник в pion-форму `engine.Session` = фактически переписать媒 медиастек с нуля.

## Что потребовалось бы для настоящей попытки (вне возможностей этой сессии)

1. Реальный аккаунт MAX (номер телефона) и живой звонок в браузере web.max.ru.
2. Перехват WebSocket-сигналинга этого звонка и полный реверс SDP/ICE-конверта внутри
   `transmit-data` + протокола OK.ru vchat.
3. Механизм персистентной «комнаты» — которого у MAX, судя по API, нет; пришлось бы держать
   постоянный исходящий звонок аккаунта-exit и как-то заводить в него клиента (у клиента тоже
   нужен аккаунт MAX) — модель, которая ломает UX carrier-а.
4. Реализация нового engine `internal/engine/max/` (pion) + провайдера с постоянной phone-auth
   сессией, поднятие `OLCRTC_REF`, снятие гейтов.

Плюс операционные/правовые риски: привязка exit-а к телефонному аккаунту в мессенджере VK,
вероятный бан аккаунтов за автоматизацию, вопросы ToS.

## Рекомендация

MAX как carrier — тупик при нынешних вводных. Задачу «маскировка под видеозвонок» уже закрывают
три рабочих carrier-а (Telemost / WbStream / Jitsi), у которых есть и комната по ссылке, и
не-телефонный вход. Предлагается **не развивать MAX** до появления (а) публичного полного реверса
in-call сигналинга MAX и (б) у MAX модели гостевого входа по ссылке. Гейты в коде оставлены как
защита: любая попытка назначить `max` отклоняется с указателем на этот документ.

## Состояние кода (staged/gated, безопасно)

- backend `olcconf` — per-login provider-override generic (хранит любую строку), правок не требует;
- backend panel/admin — `max` **отклоняется** гейтом `olcMaxCarrierReady=false` с понятным
  сообщением и ссылкой на этот док; пункт «MAX» в выпадающем списке панели убран;
- ops `olcrtc-room.sh` — `max` отклоняется за `MAX_READY=0`;
- app (ТВ) — helper `olcrtcCarrierLabel()` содержит инертную ветку «через MAX» на случай будущей
  поддержки (сейчас никогда не срабатывает — бэкенд не отдаёт `provider:"max"`).
