# Runbook — "Server 1 died" (control-plane recovery)

Server 1 (`wapmixx.ru` / `194.48.141.106`) is the **control-plane SPOF**: it runs
`maestro-panel` (the app subscription + provisioning backend), 3x-ui/Xray (VLESS, the
customer base), the primary Telegram bot, and the nginx TLS front (`:8911 → :8910`).
This runbook restores it from the off-site encrypted backups.

> Data-plane note: an S1 outage does **not** drop already-connected users. The app
> overwrites its local config only after a successful `/sub` fetch **and**
> `Libbox.checkConfig` (see `UpdateProfileWork.UpdateTask.doWork`), so when `/sub` is
> down it keeps the last-good config and the tunnel stays up; multi-protocol `urltest`
> fails over within the data plane. What's lost until recovery: **new provisioning,
> renewals propagating, and key-rotation delivery** — not connectivity for existing devices.

---

## What is backed up (and what isn't)

Hourly + on-change, `maestro-backup.sh` tars these and uploads them **gpg-encrypted** to the
**private** bucket `s3://maestro-backups/cp/<host>/` (Yandex Object Storage, `ru-central1`):

- `/var/lib/maestro/customers.json` — the unified customer base (**irreplaceable**)
- `/var/lib/maestro/orders.json` — purchase/order ledger (**irreplaceable**)
- `/etc/maestro-panel.env` — panel config **+ all secrets** (admin token, XUI token, S2 password, VLESS Reality keys, bot token)
- `/etc/x-ui/x-ui.db` — 3x-ui sqlite: VLESS clients, inbounds (**irreplaceable**)

**Not backed up (reconstructable):** the `maestro-panel` binary (rebuild from the repo), nginx
vhost + systemd units (recreate per this runbook), TLS cert (re-issue via certbot), the S2
protocol configs (hy2/caddy/anytls — regenerated by re-provisioning from `customers.json`),
the OTA `update/` dir (regenerated by `maestro-update-mirror.timer`).

## Keys you need (BOTH live OFF Server 1)

Recovery requires two secrets that deliberately do **not** live on S1, so they survive its death:

1. **Yandex S3 static key** (to read the bucket) — held by the **owner** (on the live S1 it sits in
   `/root/.aws`, profile `yc`). Keep an offline copy.
2. **The backup gpg PRIVATE key** (to decrypt) — stored on **Server 2** at
   `/root/maestro-backup-PRIVATE.asc` (0600) and imported into S2's gpg keyring; plus the owner's
   offline copy. S1 holds only the **public** key, so a lost/compromised S1 cannot decrypt its own
   backups. Fingerprint `1E93609C39AC27C46F025775A07E4D46B75ADEF1`.

## Recovery steps

### 0. Provision a new box
Ubuntu 24.04, ≥2 GB. Install: `awscli` (or `pipx install awscli`), `gpg`, `nginx`, Go (to build
the panel) or copy a prebuilt `maestro-panel` binary, and 3x-ui.

### 1. Fetch + decrypt the latest backup
On Server 2 (which holds the gpg private key) or any trusted box with the Yandex S3 key + the gpg
private key:

```bash
# configure the yc profile with the owner's Yandex S3 key first (~/.aws)
EP="--endpoint-url=https://storage.yandexcloud.net"
LATEST=$(aws --profile yc $EP s3 ls s3://maestro-backups/cp/ubuntu24/ | sort | tail -1 | awk '{print $4}')
aws --profile yc $EP s3 cp "s3://maestro-backups/cp/ubuntu24/$LATEST" /tmp/cp.tgz.gpg

gpg --import /root/maestro-backup-PRIVATE.asc      # if not already in the keyring
gpg --decrypt /tmp/cp.tgz.gpg > /tmp/cp.tgz
mkdir -p /tmp/cp && tar -xzf /tmp/cp.tgz -C /tmp/cp
# restored tree: /tmp/cp/var/lib/maestro/*.json, /tmp/cp/etc/maestro-panel.env, /tmp/cp/etc/x-ui/x-ui.db
```

### 2. Restore state onto the new box
```bash
install -D -m600 /tmp/cp/var/lib/maestro/customers.json /var/lib/maestro/customers.json
install -D -m600 /tmp/cp/var/lib/maestro/orders.json   /var/lib/maestro/orders.json
install -D -m600 /tmp/cp/etc/maestro-panel.env         /etc/maestro-panel.env
install -D -m600 /tmp/cp/etc/x-ui/x-ui.db              /etc/x-ui/x-ui.db
shred -u /tmp/cp.tgz; rm -rf /tmp/cp /tmp/cp.tgz.gpg    # wipe plaintext
```

### 3. Reinstall services
- **3x-ui**: install x-ui, stop it, drop in the restored `/etc/x-ui/x-ui.db`, start. Verify the
  VLESS inbounds + clients are present (panel `:2053`).
- **maestro-panel**: build from the repo (`go build -o /usr/local/bin/maestro-panel ./backend/cmd/maestro-panel`)
  or restore a kept binary. Recreate `/etc/systemd/system/maestro-panel.service`
  (`EnvironmentFile=/etc/maestro-panel.env`, `ExecStart=/usr/local/bin/maestro-panel`). `daemon-reload`.
- **nginx**: recreate the vhost `:8911 ssl (server_name wapmixx.ru) → 127.0.0.1:8910`; re-issue the
  TLS cert (`certbot --nginx -d wapmixx.ru`).
- **Telegram bot** (`vpnbot`): restore `/root/vpn_bot` (its own git/.env) if the bot also moved.

### 4. Repoint DNS (the critical step for the app)
The app hard-codes `https://wapmixx.ru:8911`. Update the **`wapmixx.ru` A record** to the new box IP.
Recovery latency is dominated by DNS TTL — **pre-lower the `wapmixx.ru` TTL to 300s now** so a future
cutover is fast.

### 5. Start + verify
```bash
systemctl enable --now x-ui maestro-panel nginx
curl -s https://wapmixx.ru:8911/healthz                 # -> ok
# pick a known sub_token from customers.json and confirm the sub renders 5 outbounds:
curl -s "https://wapmixx.ru:8911/sub/<token>" | grep -o '"type":"[a-z2]*"' | sort -u
```
Then claim/activate a throwaway login end-to-end and connect a test device.

### 6. Re-converge the data plane
`maestro-panel` runs `ReconcileExpiries()` at startup + every 15 min and re-syncs S2 (hy2/naive/
anytls) from the restored `customers.json`, so the S2 protocol configs self-heal. Confirm S2
services are active (`caddy`, `hysteria-server`, `sing-box-anytls`).

## RPO / RTO
- **RPO**: ≤ 1 h for `x-ui.db`/`env` (hourly timer); **near-zero** for `customers.json`/`orders.json`
  (a `systemd .path` unit backs up on every change).
- **RTO**: box setup (~20–40 min) + DNS propagation (= TTL). Pre-lowering the TTL is the cheapest RTO win.

## Periodic drill
Quarterly: download the latest object, decrypt with the S2 key, `tar -tzf` it, and diff a customer
count against live. (The full decrypt chain was verified on 2026-06-21 when this system was built.)
